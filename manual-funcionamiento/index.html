    Manual de funcionamiento — NovaMarket

    Documento técnico-operativo detallado que describe el funcionamiento del sistema, sus componentes, flujos, integraciones y arquitectura.

    1. Propósito del sistema

    NovaMarket es una plataforma de operación comercial para retail con múltiples cajas y catálogo centralizado. Integra:

    • Frontend Angular (market-ng)
    • API Gateway + microservicios Spring Boot
    • Identidad Keycloak (OIDC)
    • Mensajería Kafka (eventos asíncronos)
    • Observabilidad Prometheus / Grafana / Loki
    • Persistencia PostgreSQL por microservicio

    Objetivo: Soportar operaciones concurrentes en múltiples puntos de venta con escalado horizontal.

    5. Flujo funcional: venta en caja

    5.1 Diagrama de secuencia

    sequenceDiagram
    participant U as Usuario (Cajero)
    participant UI as Angular POS
    participant GW as Gateway
    participant KC as Keycloak
    participant V as ms-venta
    participant A as ms-articulo
    participant R as ms-rubro
    participant P as ms-pago
    participant K as Kafka
    participant DB as PostgreSQL

    U->>UI: Escanea producto
    UI->>GW: GET /api/v1/articulos/codigo/{codigo} (JWT)
    GW->>KC: Valida JWT
    KC-->>GW: Token válido
    GW->>A: Forward request
    A->>DB: SELECT producto
    DB-->>A: Producto
    A-->>UI: Producto

    U->>UI: Agrega más productos
    U->>UI: Confirma pago

    UI->>GW: POST /api/v1/ventas (JWT, items, pago)
    GW->>V: Forward request
    V->>V: validarPago()
    V->>A: validar stock (Feign)
    A->>R: obtener rubro (Feign + CB)
    R-->>A: Rubro (o fallback)
    A-->>V: Stock OK
    V->>DB: BEGIN TRANSACTION
    V->>DB: INSERT orden
    V->>DB: INSERT orden_detalle
    V->>P: registrar pago (Feign)
    P->>DB: INSERT pago
    P-->>V: Pago registrado
    V->>A: descontar stock (Feign)
    A->>DB: UPDATE stock
    A-->>V: Stock descontado
    V->>DB: COMMIT
    V->>K: publicar orden-eventos (async)
    V-->>GW: VentaResponse + numeroBoleta
    GW-->>UI: VentaResponse
    UI->>UI: generar boleta PDF
    UI->>U: Mostrar boleta

    5.2 Pasos detallados

    1. Escaneo de producto:
    2. Frontend llama a GET /api/v1/articulos/codigo/{codigo}
    3. Gateway valida JWT con Keycloak
    4. ms-articulo busca producto en PostgreSQL

    5. Retorna nombre, precio, stock

    6. Construcción del carrito:

    7. Frontend mantiene carrito en memoria (PosCartService)
    8. Valida stock antes de agregar

    9. Calcula subtotal automáticamente

    10. Confirmación de pago:

    11. Frontend envía POST /api/v1/ventas con:

      • Items (productoId, cantidad)
      • Medio de pago (EFECTIVO/TARJETA/YAPE)
      • Monto recibido (efectivo)
      • Tipo tarjeta / código autorización (tarjeta)
      • Código operación Yape (Yape)
      • Cajero username

    12. Procesamiento en ms-venta:

    13. Valida datos de pago
    14. Itera items y valida stock en ms-articulo
    15. Calcula subtotal, descuento, total
    16. Genera número de boleta: NM-00000001

    17. Persiste orden y detalle en PostgreSQL

    18. Registro de pago:

    19. Llama a ms-pago vía Feign
    20. ms-pago persiste pago en PostgreSQL

    21. Retorna código autorización, referencia transacción

    22. Descuento de stock:

    23. Por cada item, llama a ms-articulo
    24. ms-articulo actualiza stock en PostgreSQL

    25. Si stock insuficiente, rollback transacción

    26. Publicación de evento Kafka:

    27. Publica orden-eventos con:
      • ordenId
      • total
      • estado
      • timestamp
    28. ms-pago consume evento (asíncrono)

    29. Si Kafka no está disponible, la venta no falla (try-catch)

    30. Generación de boleta:

    31. Frontend genera PDF térmica 55mm
    32. Descarga archivo y abre diálogo de impresión

    6. Seguridad

    6.1 Flujo de autenticación

    sequenceDiagram
    participant U as Usuario
    participant UI as Angular
    participant KC as Keycloak
    participant GW as Gateway
    participant MS as Microservicio

    U->>UI: Click "Entrar con Keycloak"
    UI->>KC: Redirect a /auth
    KC->>KC: Login usuario/password
    KC->>UI: Redirect con code
    UI->>KC: Token exchange (code → JWT)
    KC-->>UI: Access Token (JWT)
    UI->>UI: Guardar token en localStorage

    U->>UI: Navega a /pos
    UI->>GW: GET /api/v1/articulos (Authorization: Bearer JWT)
    GW->>KC: Validar JWT (introspection o local)
    KC-->>GW: Token válido + roles
    GW->>MS: Forward request con roles
    MS-->>UI: Response

    6.2 Configuración Keycloak

    Realm: novamarket

    Client: market-ng - Access Type: public - Valid Redirect URIs: http://localhost:4200/* - Web Origins: http://localhost:4200

    Roles: - admin — Acceso completo - supervisor — Gestión sin eliminar - cajero — Solo ventas y consulta

    6.3 Validación en microservicios

    Gateway: - Valida JWT en cada request - Extrae roles del token - Agrega headers X-User-Roles al request

    ms-articulo: 

    @PreAuthorize("hasRole('admin') or hasRole('supervisor')")
@PostMapping
public ResponseEntity<ProductoResponse> create(@Valid @RequestBody ProductoRequest request) {
    // Solo admin y supervisor pueden crear
}

    ms-venta: - Valida que el cajero tenga rol cajero, supervisor o admin - Verifica que el username del token coincida con el cajero de la venta

    7. Comunicación entre servicios

    7.1 OpenFeign

    Cliente Feign en ms-venta: 

    @FeignClient(name = "MS-ARTICULO", url = "${ms-articulo.url:}")
public interface ProductoVentaClient {
    @GetMapping("/api/v1/articulos/{id}")
    ProductoVentaDto obtenerProducto(@PathVariable Integer id);

    @PostMapping("/api/v1/articulos/{id}/descontar-stock")
    void descontarStock(@PathVariable Integer id, @RequestParam Integer cantidad);
}

    Configuración: 

    feign:
  circuitbreaker:
    enabled: true
  client:
    config:
      default:
        connectTimeout: 5000
        readTimeout: 5000

    7.2 Circuit Breaker (Resilience4j)

    Configuración en ms-articulo: 

    resilience4j:
  circuitbreaker:
    instances:
      rubroService:
        failure-rate-threshold: 50
        wait-duration-in-open-state: 10s
        sliding-window-size: 10

    Implementación: 

    @CircuitBreaker(name = "rubroService", fallbackMethod = "rubroFallback")
public RubroResponse obtenerRubro(Integer id) {
    return rubroClient.obtenerRubro(id);
}

private RubroResponse rubroFallback(Integer id, Exception ex) {
    log.warn("Fallback para rubro {}: {}", id, ex.getMessage());
    return RubroResponse.builder().id(id).nombre("Sin categoría").build();
}

    8. Escalado y concurrencia (multi-caja)

    8.1 Escalado horizontal

    Levantar segunda instancia de ms-venta: 

    cd services\ms-venta
# Terminal 1
mvn spring-boot:run
# Terminal 2
set SERVER_PORT=19052
mvn spring-boot:run

    Verificar en Eureka: http://localhost:18761 → Debe ver 2 instancias de MS-VENTA

    8.2 Balanceo de carga

    Gateway usa round-robin por defecto entre instancias registradas.

    Configuración: 

    spring:
  cloud:
    loadbalancer:
      ribbon:
        enabled: false  # Spring Cloud LoadBalancer en lugar de Ribbon

    8.3 Demo multi-caja

    Pasos: 1. Levantar 2 instancias de ms-venta 2. Abrir Chrome (cajero1) 3. Abrir Incógnito o Edge (cajero2) 4. Ambos escanean el mismo producto 5. Ambos cobran casi al mismo tiempo 6. Stock se descuenta correctamente (mismo PostgreSQL)

    Resultado: - Gateway balancea peticiones entre las 2 instancias - Stock compartido (mismo contador en BD) - Simula varias cajas en una tienda real

    ms-articulo (articulo_db): - articulo (id, nombre, rubro_id, precio, stock, stock_minimo, codigo_barras, created_at, updated_at)

    ms-venta (venta_db): - orden (id, cajero_username, subtotal, descuento, total, estado, medio_pago, monto_recibido, vuelto, numero_boleta, fecha_venta) - orden_detalle (id, orden_id, producto_id, producto_nombre, cantidad, precio_unitario, subtotal)

    ms-pago (pago_db): - pago (id, orden_id, total, medio_pago, monto_recibido, tipo_tarjeta, codigo_autorizacion, codigo_operacion, moneda, referencia_transaccion, fecha_pago)
